📋 COPILOT AGENT KNOWLEDGE LAYER — REUSABLE PLAYBOOK  v2.9
═══════════════════════════════════════════════════════════════

Bootstrap an AI-native knowledge layer for any codebase so that GitHub
Copilot (and new devs) can navigate, code, debug, and review with full
context.

Prerequisite: GitHub Copilot Chat in AGENT MODE (not Ask/Edit).

How to use:
- Copy each prompt into Copilot Chat, run phases in order (0→6).
- Replace «double angle brackets» with your repo's values. DELETE examples.
- Designed for end-to-end single session or split via anchor-file resume.


═══════════════════════════════════════════════════════════════
🚨 DEEP ANALYSIS MANDATE (applies to ALL phases — never skip)
═══════════════════════════════════════════════════════════════

This playbook produces a KNOWLEDGE LAYER — not a summary layer. Every
phase must go DEEP. If you find yourself skimming, listing file names
without reading contents, or producing thin output, you are doing it
wrong. Stop and redo.

SOURCE READING — DEPTH OVER BREADTH:
  - NEVER describe a file you haven't read. Open it, read 50+ lines
    of method bodies — not just imports, class signatures, or comments.
  - For every claim you write (class name, method call, config key,
    error code), you MUST have seen it in actual source code.
  - When a flow spans multiple files (controller → BL → impl), READ
    EACH HOP. Don't infer what the next layer does from its name.
  - Read METHOD BODIES. The valuable content is inside conditionals,
    catch blocks, null checks, retry loops, and config lookups —
    not in class declarations or using statements.
  - If a file is 500+ lines, read the key methods (entry points,
    error handlers, state transitions) — don't skip to the next file.

OUTPUT QUALITY — NO THIN CONTENT:
  - Every table cell must contain REAL data from source code, not
    placeholders like "TBD", "see code", or "various".
  - Every workflow flow section must have ≥5 steps with real method
    names and file paths. A 3-step flow means you didn't read deep
    enough.
  - Every error scenarios table must have ≥3 rows with real error
    codes found in catch blocks, not generic "invalid input" entries.
  - Every config dependencies table must list ACTUAL config keys read
    from config files (e.g., AppConfig, appsettings), not guessed property names.
  - Code exemplars must cite REAL file paths and show REAL code
    patterns — not generic templates.

VERIFICATION — PROVE YOUR WORK:
  - When you write "method X calls Y", you must have seen the call
    site. Cite the file and approximate line range.
  - When you write "config key Z controls behavior W", you must have
    seen where Z is read in code AND where it affects logic.
  - If you can't verify a claim, DON'T WRITE IT. An empty cell is
    better than a hallucinated one.
  - After completing any doc, re-read your output and ask: "Could I
    debug a production issue using ONLY this doc?" If no → add detail.

COMPLETENESS — NO PHASE SHORTCUTS:
  - Every phase has a defined OUTPUT format (table, file, list).
    Produce the FULL output, not a summary of what you would produce.
  - Quality loops and follow-ups are NOT optional. They exist because
    the first pass always misses things.
  - If a phase says "per domain" or "per doc", run it for EVERY domain
    and EVERY doc — don't pick 2 of 10 and call it done.
  - If you hit context limits mid-phase, STOP and tell the user to
    resume with SCOPE narrowed — don't silently produce shallow output
    for the remaining items.

ANTI-PATTERNS (if you catch yourself doing these, stop and redo):
  - Listing 20 files but only reading 3 → Read all or split into batches
  - Writing "handles errors" without listing which error codes → Read catch blocks
  - Copying class XML doc comments as your documentation → Read the method body
  - Producing a workflow with steps like "processes the request" → Name the method
  - Skipping the follow-up/quality loop → It's mandatory, not optional
  - Saying "similar to above" or "follows same pattern" → Document it explicitly
  - Generating config tables from parameter names alone → Read where they're used

PHASE 0: Discovery Manifest
═══════════════════════════════════════════════════════════════

Prompt:

  Create a discovery manifest — the anchor file for all subsequent phases.

  Paths: DOCS_ROOT={DOCS_ROOT}, SKILLS_ROOT={SKILLS_ROOT}
  Use these for ALL file creation in every phase.

  SCOPE: «blank for full repo, or specify folders»

  1. Scan repo top-down. Per major folder record: path, purpose, file
     count, primary language, key entry points.

  2. Detect existing knowledge assets:
     - .github/copilot-instructions.md — Y/N, line count
     - .github/copilot-memory.md — Y/N, line count
     - SKILL.md files — list with paths
     - docs/ folders — list with file counts
     - README.md with architecture info — list
     - Existing workflow/API/runbook docs

  3. Detect tech stack from actual files (don't guess):
     Languages, frameworks, test frameworks, CI/CD.

  3B. DOMAIN DISCOVERY ALGORITHM (advisory — use when grouping is ambiguous):

     Most codebases have obvious domain boundaries. Use this algorithm
     as a TIEBREAKER when it's unclear whether controllers should be
     merged, split, or standalone. Also useful for consistency when
     multiple people run the playbook on the same repo.

     STEP 1 — FIND ENTRY POINTS:
     Scan for the framework's HTTP entry points:
       | Framework | Entry Point Pattern | Downstream Pattern |
       |-----------|--------------------|-----------------------|
       | ASP.NET Core | *Controller.cs | *Impl.cs, *Service.cs, *Provider.cs |
       | Spring Boot | *Controller.java | *Service.java, *Repository.java |
       | FastAPI / Django | routes/*.py, views.py | services/*.py |
       | Express / NestJS | *.controller.ts | *.service.ts |
       | Go (Chi/Gin) | *_handler.go | *_service.go |
     Each entry point file = candidate domain.

     STEP 2 — GROUP BY SHARED DOWNSTREAM:
     For each entry point, trace 1-2 levels down (BL → impl/service).
     If 2+ controllers share the SAME impl/service class → MERGE into
     one domain named after the shared impl.
       Example: ProductController + InventoryController both call
       ProductService.cs → domain = "Products & Inventory"

     STEP 3 — SPLIT BY ACTION DIVERGENCE:
     If one controller's actions route to 2+ DIFFERENT impl classes
     with unrelated logic → SPLIT into separate domains.
       Example: OrderController routes CRUD to OrderService
       but cancellation to OrderCancellationService → 2 domains.
       Heuristic: if the impl files share <30% of their dependencies
       (config reads, entity types, telemetry classes), they're separate.

     STEP 4 — NAME THE DOMAIN:
     Priority order:
       a. Resource type (orders, payments, users, products)
       b. Shared impl class name (minus "Impl" suffix)
       c. Controller name (minus "Controller" suffix)

     STEP 5 — HANDLE NON-CONTROLLER DOMAINS:
       - Design patterns (cross-region proxy, plugin architecture) →
         domain = the pattern name, doc type = "design"
       - Shared infrastructure (auth, telemetry, config) →
         domain = reference doc, not workflow doc, doc type = "reference"
       - Background workers/jobs → domain per worker type

     STEP 6 — OUTPUT: DOMAIN ROSTER TABLE
       | # | Domain Name | Entry Points | Shared Impl | Doc Type | Workflow Doc |
       Doc Type = "workflow" (has API surface) or "design" (internal
       pattern) or "reference" (shared infrastructure)
       Workflow Doc = relative path to the doc file, e.g., `workflows/01_Orders.md`
       (leave blank for domains not yet documented — Phase 2 fills these in)

     Save the domain roster in the Discovery Manifest under
     "Identified Domains" using this table format.

  4. Save to `{DOCS_ROOT}/Discovery_Manifest.md`:
     - Repo Map (folder → purpose table)
     - Existing Knowledge Assets
     - Tech Stack Summary
     - Identified Domains (features to document in Phase 2)
     - Last Updated: «date»

  This is your ANCHOR — re-read its "Repo Map" and "Identified Domains"
  at the start of every subsequent phase. Update it if you discover new
  domains later.

  OUTPUT: Create file. List: file path, domains identified, assets found.
  Proceed to Phase 1.

  NON-NEGOTIABLE RULES:
  - Section numbering MUST use `## N.` format (e.g., `## 1. Overview`)
  - For background workers: use "Key Components" for §2, "Key Workers" for §3
  - Extra sections (§12+) are encouraged but §1-§11 are required
  - Every workflow doc MUST include at least one mermaid sequence diagram in §5
  - Every workflow doc MUST include a mermaid graph TD dependency DAG in §11

  **Mermaid sequence diagram** — every §5 MUST include a ```mermaid block:
    ```mermaid
    sequenceDiagram
      participant Client
      participant Controller
      participant BL as Business Logic
      participant Impl as VaultImpl
      participant Plugin
      Client->>Controller: DELETE /orders/{id}
      Controller->>BL: DeleteOrderInternal()
      BL->>Impl: ProcessOrderDeletion()
      Impl->>Plugin: DoDeleteOrderOperation()
      Plugin-->>Impl: AsyncStatus{Succeeded}
      Impl-->>BL: Response(200 OK)
    ```
    Rules for Mermaid diagrams:
    - Use REAL method names from source code (not generic "process request")
    - Show decision points with alt/opt blocks for branching logic
    - Include error paths (catch blocks, fallbacks) as alt branches
    - Show config checks that change flow direction
    - Minimum 4 participants for any cross-layer flow
    - For complex flows, split into multiple diagrams (happy path + error path)

  Use real class/method names and file paths. Number sequentially (01_, 02_...).

PHASE 2C: Architecture Decision Records (ADRs)
═══════════════════════════════════════════════════════════════

Capture the WHY behind design decisions. Workflow docs explain HOW code
works; ADRs explain WHY it was built that way. This prevents re-debates,
helps new devs understand trade-offs, and preserves institutional knowledge.

Prompt:

  Re-read "Repo Map" + "Identified Domains" from
  `{DOCS_ROOT}/Discovery_Manifest.md`.

  Create the ADR system in `{DOCS_ROOT}/adr/`:

  STEP 1 — Create ADR template and index:

  Create `{DOCS_ROOT}/adr/README.md`:
  ────────────────────────────────────────────────────────────
  # Architecture Decision Records

  > Each ADR captures a significant design decision: the context,
  > the decision made, the alternatives considered, and the consequences.
  > ADRs are numbered sequentially and NEVER deleted — superseded ADRs
  > are marked with status "Superseded by ADR-NNNN".

  ## Index

  | ADR | Title | Status | Date |
  |-----|-------|--------|------|
  | 0001 | ADR Template | Accepted | «date» |

  ## How to Add a New ADR
  1. Copy `0001-adr-template.md`
  2. Number sequentially (next: NNNN)
  3. Fill in all sections — especially Alternatives Considered
  4. Set status to "Proposed" until reviewed, then "Accepted"
  5. Update this README index
  ────────────────────────────────────────────────────────────

  Create `{DOCS_ROOT}/adr/0001-adr-template.md`:
  ────────────────────────────────────────────────────────────
  # ADR-0001: «Title»

  **Status:** Accepted | Proposed | Deprecated | Superseded by ADR-NNNN
  **Date:** «YYYY-MM-DD»
  **Author:** «name or team»
  **Related:** «workflow doc, config property, or code path»

  ## Context
  What is the problem or situation that prompted this decision?
  What constraints exist?

  ## Decision
  What was decided? Be specific — name the pattern, config, class,
  or approach chosen.

  ## Alternatives Considered
  | Alternative | Pros | Cons | Why Rejected |
  |-------------|------|------|-------------|
  | «option A» | ... | ... | ... |
  | «option B» | ... | ... | ... |

  ## Consequences
  - **Positive:** What does this enable?
  - **Negative:** What trade-offs were accepted?
  - **Risks:** What could go wrong?

  ## References
  - Source files: «paths»
  - Workflow doc: «link»
  - Related ADRs: «links»
  ────────────────────────────────────────────────────────────

  STEP 2 — Mine existing code for architectural decisions:

  Read the workflow docs from Phase 2 and source code to discover
  decisions that should be recorded. Look for:
  - NON-OBVIOUS PATTERNS: "Why does delete check soft-delete before
    force-delete?" "Why is there a separate plugin per datasource type?"
  - CONFIG-DRIVEN BEHAVIOR: "Why is this behind a feature flag?"
    "Why does this config default to false?"
  - ERROR HANDLING CHOICES: "Why does this catch block allow cleanup
    on force purge?" "Why swallow this specific error?"
  - ARCHITECTURE BOUNDARIES: "Why is business logic in VaultImpl
    instead of Controller?" "Why a plugin architecture for datasources?"
  - CROSS-COMPONENT CONTRACTS: "Why does Fabric WF use async status
    polling instead of callbacks?"
  - SECURITY DECISIONS: "Why is MUA required for delete but not create?"
  - MIGRATION/COMPATIBILITY: "Why does this brownfield check exist?"
    "Why support both v1 and v2 API versions?"

  For each discovered decision, check:
  - Is the RATIONALE documented anywhere (comments, PR descriptions, docs)?
  - If YES → extract and formalize into an ADR
  - If NO → infer from code context, mark as "Inferred — verify with team"

  STEP 3 — Create initial ADRs:

  Create 5–10 ADRs for the most significant decisions discovered.
  Start numbering from 0002 (0001 is the template).
  Prioritize decisions that:
  - New developers would ask "why?" about
  - Have non-obvious trade-offs
  - Are config-controlled (flag could be changed, need to know why it's set)
  - Span multiple components or workflows

  Include Mermaid diagrams where they clarify the decision:
  - Use flowcharts for decision trees / branching logic
  - Use sequence diagrams for cross-component interaction choices

  Example Mermaid in ADR:
  ```mermaid
  flowchart TD
    A[Delete BI Request] --> B{Soft Delete Enabled?}
    B -->|Yes| C[Soft Delete Flow]
    B -->|No| D{Force Delete?}
    D -->|Yes| E[Direct Cleanup]
    D -->|No| F[Normal Purge via Fabric WF]
  ```

  - In each ADR's "References" section, link to relevant workflow docs

  VERIFICATION: Every ADR must cite at least one source file where the
  decision is implemented. No ADR should describe a decision that can't

── 4E: Curiosity Monkey (Autonomous Documentation Stress Test) ──

Recommended. Runs a fully autonomous code→docs→judge→fix loop to find
gaps that structured validation misses. NO human in the loop after setup.

Also available as standalone: `docs/demos/Curiosity_Monkey_Prompt.md`

How it works:

  The 🐵 Curiosity Monkey has THREE passes:

  Pass 0 — DISCOVERY: Scans all controller source files, diffs against
  existing workflow docs, and creates new docs for undocumented
  controllers. This ensures no controller is invisible to the audit.

  Pass 1 — BREADTH: Reads SOURCE CODE first, forms questions about
  interesting behavior (edge cases, config, error handling, hidden
  logic), then searches DOCS to answer its own questions. Covers
  ALL domains with 2-3 questions each.

  Pass 2 — DEPTH: Drills into weak domains from Pass 1. Stays in
  one domain until it's solid. Gaps cluster — follows up on each
  gap with deeper related questions.

  It judges coverage (✅ / ⚠️ / ❌) and auto-fixes gaps via doc
  self-healing rules in copilot-instructions.md.

  The monkey is both questioner and answerer — same Copilot instance,
  with phase-ordering guardrails to prevent reading docs before forming
  questions. No agent-to-agent chaining needed.

  - If ✅ 70–89%: "Docs need work. Run another round on ❌ domains."
  - If ✅ < 70%: "Significant gaps. Consider re-running Phase 2 and 4."

AFTER THE FINAL SCORECARD, update doc_registry.md:
  1. Update the "Last audit" date and add a summary line for this run
     (e.g., "Curiosity Monkey — N questions, deep, all domains").
  2. For each doc that was updated/created during the run:
     - Update "Last Verified" date in the Documented Controllers table

RULES:
  - CODE FIRST, ALWAYS: The monkey is a REVERSE VALIDATION tool. You read
    SOURCE CODE to discover behavior, then check if docs cover it. You
    NEVER start from docs and check if code matches. Every question MUST
    originate from something you observed in actual source code.
  - DISCOVER FIRST: If discovery mode = full, Pass 0 runs before any
    audit. Find ALL undocumented controllers, create docs for them.
  - BREADTH SECOND: Pass 1 MUST touch EVERY domain (including new
    docs from Pass 0). No domain left unvisited.
  - THEN GO DEEP: Pass 2 drills into weak domains from Pass 1.
    Stay in one domain until it's solid. Don't scatter.
  - FOLLOW THE GAPS: After finding a gap (especially in Pass 2),
    explore the SAME area deeper — related configs, error paths,
    callers, telemetry, edge cases in the same file/method.
  - BUDGET SPLIT: ~30% of total questions for Pass 1 (breadth),
    ~70% for Pass 2 (depth). E.g., 20 Qs total = 6 breadth + 14 depth.
  - NEVER read docs before forming a question. Code first, docs second.
  - You are BOTH questioner and answerer. Never wait for human input.
  - Run the full ASK → ANSWER → JUDGE → FIX loop for each question
    before forming the next.
  - Mix question types within each domain (FLOW, CONFIG, ERROR,
    TELEMETRY, EDGE CASE, etc.) but bias toward types that found gaps.
  - Prefer non-obvious behavior: branching logic, fallbacks, retry
    mechanisms, cross-component calls, hidden config dependencies.
  - Deep difficulty = multi-hop questions spanning multiple files.
    Follow calls 2-3 levels deep (controller → BL → VaultImpl → plugin).
  - When fixing gaps, follow doc self-healing rules in
    copilot-instructions.md (update existing docs first, don't create
    duplicates, update indexes).
  - Always cite the source file/method that prompted the question.
  - Print progress continuously — the user is watching, not participating.

ANTI-SHORTCUT GUARDRAILS (mandatory — never skip these):

  CODE READING DEPTH:
  - You MUST read at least 50 lines of actual source code per question.
    Reading file names, class signatures, or using statements is NOT enough.
  - Read METHOD BODIES — the interesting behavior is inside conditionals,
    catch blocks, null checks, and config lookups.
  - For deep difficulty: follow at least 2 call hops (A calls B calls C).
  - PROVE you read the code: cite specific line ranges, variable names,
    or conditional expressions in your question.

  QUESTION QUALITY:
  - NEVER ask surface-level questions like "What does this class do?"
    or "What is the purpose of this controller?"
  - Every question MUST reference a SPECIFIC code construct: a conditional
    branch, a catch block, a config read, a null check, a retry loop,
    a cross-component call, or a non-obvious constant.
  - If your question could be answered by reading the class name alone,
    it's too shallow. Dig deeper.

  DOC SEARCH DEPTH:
  - You MUST search at least 3 different doc files before declaring
    "docs silent". Search: relevant workflow doc, architecture memory,
    error code reference, glossary, and telemetry reference.
  - Use grep/search with multiple keyword variants — don't just check
    one term. Try the method name, the class name, the config key,
    AND the error code.
  - PROVE you searched: list which docs you checked and what you searched for.

  JUDGING RIGOR:
  - ✅ COVERED means docs explain the EXACT behavior you found in code,
    not just that they mention the feature in passing.
  - If docs say "handles locking" but don't explain the ActiveActive
    dynamic path resolution you found, that's ⚠️ PARTIAL, not ✅.
  - When in doubt between ✅ and ⚠️, choose ⚠️. When in doubt between
    ⚠️ and ❌, choose ❌. ERR ON THE SIDE OF FINDING GAPS.

  FIX QUALITY:
  - Every fix MUST include: what the code does, when it triggers,
    source file path, and at least one relevant detail (config key,
    error code, or edge case behavior).
  - One-sentence fixes are NOT acceptable. Minimum: a paragraph or a
    table with 3+ rows of detail.
  - Include a code reference (file + method name) in every fix.

  NO BATCHING:
  - ONE question at a time. Complete the full ASK→ANSWER→JUDGE→FIX
    loop before forming the next question.
  - NEVER combine multiple questions into one.
  - NEVER skip the ANSWER phase by saying "I already know from code."

Tips:
  - Three-pass model: Pass 0 discovers undocumented controllers (creates
    new docs), Pass 1 breadth scans ALL domains, Pass 2 depth drills.
  - Discovery mode: Use "full" on first run or after major code changes.
    Use "audit-only" for routine sweeps on existing docs.
  - Budget: Pass 0 doesn't consume the question budget. For N audit Qs
    (default 100), ~30% breadth + ~70% depth.
    E.g., 100 Qs = ~30 breadth (2-3 per domain) + ~70 depth (worst domains first).
  - Gap hunting: consecutive-skip=5, difficulty=deep to dig harder.
  - Targeted: Focus on a domain you just changed (skips Pass 0+1).
  - Pre-PR: audit-only, 3–5 deep questions on the changed domain.
  - Dry-run first: See what gaps exist before committing fixes.
  - Config verification: The monkey verifies config references against
    source code — fabricated config names were 40% of bugs in prior runs.

Exit criteria:
  - Pass 0: All controllers have a workflow doc (or section in one).
  - ✅ ≥ 90% → Docs are solid. Proceed to 4D.
  - ✅ 70–89% → Re-run on ❌ domains with more questions.
  - ✅ < 70% → Significant gaps. Consider re-running Phase 2 first.

When to run:
  - After Phase 4A–4C (initial build validation gate)
  - Pre-PR (targeted: focus on changed domain, 3–5 deep questions)
  - Periodic sweeps (catch drift as code evolves)
  - Post-feature (run on the domain you just changed)
